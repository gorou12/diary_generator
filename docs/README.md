# ドキュメントガイド

本ディレクトリ (`docs/`) は、このプロジェクトにおける **仕様の正本（source of truth）** を定義する。

## 目的

- 人間が仕様を整理・決定するための場所
- AI（Cursor等）に対する指示の基準となる資料
- 実装と仕様の乖離を防ぐ

---

## ドキュメントの優先順位

仕様に矛盾がある場合、以下の優先順位で判断する。

1. `docs/` 配下の文書（本ディレクトリ） ← 最優先
2. 実装コード（`scripts/` 等）
3. `docs_by_ai/`（現行コードの説明資料）

---

## docs_by_ai との関係

`docs_by_ai/` は、現行コードをもとにAIが生成した **現状仕様（as-is）ドキュメント** である。

- 実装の挙動を把握するための参考資料として使用する
- 仕様の決定には使用しない
- 内容が `docs/` と矛盾する場合は `docs/` を正とする

---

## 想定する運用フロー

1. 本ディレクトリ（`docs/`）で仕様を定義・更新する
2. Cursor に対して、該当ドキュメントを提示して変更指示を行う
3. 実装を更新する
4. 必要に応じて `docs_by_ai/` を更新または無視する

---

## AI（Cursor）への指示方針

AIに実装を依頼する際は、以下を明示すること：

- 参照するドキュメント（例：`docs/features/...`）
- 変更対象範囲
- 変更してはいけない範囲
- 互換性要件（URL・JSONなど）

また、以下のルールを前提とする：

- 仕様に書かれていない挙動は変更しない
- 不明点は既存コードに合わせる
- 推測で仕様を拡張しない

---

## ドキュメント構成（予定）

- `policies.md`
  - システム全体のルール・制約・設計方針
- `features/`
  - 機能単位の仕様
- `screens/`
  - 画面単位の仕様
- `json/`
  - データ構造・JSON仕様

---

## 補足

本プロジェクトは個人開発による「静的HTML生成＋Notionデータ取得」を前提とした日記システムである。

設計上の判断は、以下を重視する：

- 公開データの安全性
- URLの安定性（既存リンクの維持）
- 実装の単純さ
- 運用の容易さ
